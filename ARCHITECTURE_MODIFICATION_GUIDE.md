# 架构与设计问题修改指导文档

## 1. 目标

本文档仅用于指导当前仿真设置与绘图链路重构。

- 建立唯一的仿真设置权威来源
- 保证 UI 设置、代码设置、SPICE 命令三者绝对一致
- 处理 UI 能设置但代码不能完整表达、以及代码能表达但 UI 未覆盖的情况
- 建立仿真设置到结果元数据再到绘图行为的单一契约
- 激进删除旧设计、旧入口、旧兼容路径，不保留过时方法

本文档不写具体代码，只写执行原则与重构步骤。

---

## 2. 核心判断

### 2.1 唯一权威

仿真求解语义必须只有一个权威来源：结构化仿真命令模型。

- UI 不是独立语义源，只是编辑入口
- 代码生成的 SPICE 不是第二套语义，而是权威模型的文本表示
- 执行器不得自行补默认语义，除非该默认值已明确属于权威模型
- 绘图层不得自行猜测横轴类型、范围、对数/线性方式

### 2.2 两类设置必须严格分层

- **求解设置**：决定仿真计算结果，必须可稳定落到 SPICE 命令
- **显示设置**：决定图如何展示，不得反向影响求解命令

### 2.3 UI 与代码同时可设置的项

对于 UI 和代码都能设置的项，必须保持绝对一致。

- 两者必须映射到同一份内部模型
- 不允许 UI 存一份、代码再解释一份
- 不允许执行器根据 UI 做额外推断
- 不允许绘图层根据结果类型再覆盖这些设置的本意

### 2.4 UI 与代码不完全重叠的项

必须按下面规则处理：

- **代码可表达、UI 未覆盖**：允许存在，但必须进入统一模型，并在系统中保持可追踪
- **UI 想提供、代码无法表达**：不得伪支持；要么删除，要么明确降为纯显示设置
- **仅属于显示层的项**：不得混入求解配置模型

---

## 3. 当前重构后的目标结构

### 3.1 命令层

负责表达 `.op`、`.ac`、`.dc`、`.tran`、`.noise` 及必要全局求解参数。

要求：

- 能作为唯一求解配置来源
- 能稳定渲染为 SPICE 命令
- 能被执行器直接消费
- 能明确哪些字段已支持，哪些字段未支持

### 3.2 结果元数据层

负责把求解语义传递给显示层。

至少要能表达：

- 当前分析类型
- 横轴语义类型
- 横轴标签
- 横轴默认比例方式
- 请求范围与实际范围
- 实际执行命令文本

### 3.3 显示层

显示层只消费结果元数据与仿真结果。

- 不再猜测时间轴、频率轴、扫描轴
- 不再硬编码 AC 或噪声一定使用某种显示方式
- 不再让图表选择系统凌驾于结果语义之上

---

## 4. 必须落实的统一原则

### 4.1 配置一致性原则

- 同一设置项只能有一个内部字段
- 同一设置项只能有一个保存格式
- 同一设置项只能有一个执行解释方式
- 同一设置项只能有一个绘图默认含义

### 4.2 显示从结果出发原则

- 图表范围优先服从仿真结果的真实坐标范围
- 图表默认坐标模式服从结果元数据
- 用户显示操作只改变视图，不得回写求解语义

### 4.3 不伪兼容原则

- 不兼容旧配置语义
- 不保留旧字段的双写逻辑
- 不保留过渡性分支
- 不保留名称不同但职责重复的方法

### 4.4 激进清理原则

在重构中必须同步删除：

- 旧设置入口
- 重复配置模型
- 旧图表选择语义
- 已失效的适配函数
- 与新契约冲突的推断逻辑
- 仅为兼容旧路径而存在的多余方法

---

## 5. 重构重点

### 5.1 统一设置入口

当前“分析启用/图表启用”和“分析参数编辑”是拆裂的，后续必须收敛。

目标：

- 让分析类型、分析参数、求解级全局参数进入同一套权威配置体系
- 将图表显示偏好从求解设置中剥离

### 5.2 补全基础分析能力

本轮只聚焦基础分析：

- `.op`
- `.ac`
- `.dc`
- `.tran`
- `.noise`

其中必须优先补齐当前缺口：

- `DC` 的扫描类型与扫描语义
- `Noise` 的扫描类型与点数语义
- `Transient` 的 `UIC` 等求解级语义真实落地

### 5.3 建立结果到绘图的正式契约

重构后，绘图行为必须来自结果元数据，而不是来自显示层猜测。

重点处理：

- 横轴标签
- 横轴范围
- 横轴比例方式
- 不同分析类型下可绘制图表的合法集合

### 5.4 收敛图表系统职责

图表系统应只负责“如何显示已存在的数据”，不再负责定义仿真语义。

---

## 6. 执行顺序

### Phase 1：收敛权威模型

- 定义统一求解配置结构
- 明确 UI、保存、执行、结果使用同一份语义
- 停止新增旧配置路径上的修改

### Phase 2：重构执行器输入

- 执行器只接受统一配置语义
- 删除执行器内部散落的补丁式推断
- 补齐基础分析命令生成缺口

### Phase 3：重构结果元数据

- 将分析坐标语义写入结果对象
- 让结果对象成为绘图层唯一解释来源

### Phase 4：重构绘图入口

- 波形图与分析图统一消费结果元数据
- 删除显示层对分析类型的硬编码推断
- 删除与新契约冲突的旧逻辑

### Phase 5：清理旧设计

- 删除重复对话框职责
- 删除旧图表选择遗留路径
- 删除旧字段、旧方法、旧兼容分支

### Phase 6：闭环复核

逐项复核以下闭环：

- 设置
- 命令生成
- 结果装配
- 图表范围
- 图表默认坐标行为

只检查基础分析，不扩展高级能力。

---

## 7. 本轮修改边界

本轮必须完成：

- 基础分析的统一求解语义
- UI/代码共有设置项的一致化
- 绘图与仿真设置范围的一致化
- 旧设计和冲突路径的删除

本轮不优先扩展：

- LTspice 的高级辅助选项
- 非基础分析
- 为兼容旧数据而保留的复杂过渡逻辑

---

## 8. 完成标准

当以下条件同时成立时，视为本轮重构完成：

- UI 与代码对同一设置项不再存在双重语义
- `.op/.ac/.dc/.tran/.noise` 的关键设置能稳定落到命令
- 结果对象能明确表达绘图所需的坐标语义
- 绘图范围与仿真设置严格一致
- 显示层不再猜测求解语义
- 旧设计、旧入口、旧兼容路径已被删除
