# 仿真领域 Web 化视觉重构设计文档

## 1. 文档目标

本文档用于指导 `circuit_design_ai` 的仿真领域视觉显示重构。

本轮重构不是对现有仿真功能做局部美化，也不是继续在 Qt Widgets 上追加样式补丁，而是要建立一套新的、统一的、权威的 Web 化仿真显示架构，使仿真领域在保持现有核心能力正确的前提下，完成以下目标：

- 将仿真结果显示、波形交互、分析信息与导出展示统一迁移到 Web 技术栈
- 保持仿真执行、仿真结果存储、撤回机制三者与显示层彻底分离
- 保留并升级当前已经存在的能力，包括图表绘制、信号选择、框选放大、光标、测量、添加至对话、自适应布局等
- 建立单一权威的前端显示实现，避免 Qt Widget 页面与 Web 页面长期双轨并存
- 在重构完成后，从代码层面彻底删除旧显示设计、旧入口、旧兼容路径

本文档不写任何具体代码，只写：

- 当前真实架构事实
- 重构边界与核心判断
- 目标架构
- 分阶段执行步骤
- 关键实现细节
- 必须彻底根除的旧设计
- 验收标准与闭环检查项

---

## 2. 重构边界与非目标

### 2.1 本轮重构边界

本轮聚焦于仿真领域的显示与交互表现层，范围包括：

- 仿真结果主面板的整体布局
- 状态提示与空状态显示
- 指标摘要显示
- 分析图表显示
- 波形图显示
- 分析信息、原始数据、输出日志、导出信息等结果面板
- 波形交互能力，包括信号选择、框选放大、视口重载、光标、测量、图例、网格、适配大小
- 添加到对话的显示入口与交互入口
- Web 技术栈下的资源加载、桥接协议与状态下发方式

### 2.2 明确不做的事

本轮不应演变为以下方向：

- 不重写仿真求解器
- 不重写仿真执行器注册与执行链路
- 不改变仿真结果文件的持久化语义
- 不把撤回机制改成仿真面板内部逻辑
- 不让前端成为仿真结果、撤回、导出或对话附件的第二权威源
- 不把整个应用外壳改成 Web
- 不为了过渡长期保留 Qt 版仿真结果界面和 Web 版界面同时存在

### 2.3 必须保留的后端权威边界

以下能力在本轮之后仍必须由后端权威维护：

- 仿真执行入口与运行生命周期
- 仿真结果文件落盘与读取
- 仿真结果对象与其元数据
- 回滚与工作区同步机制
- 波形数据多分辨率访问与缓存
- 测量计算逻辑
- 添加到对话所需的工件导出

显示层只允许消费这些权威结果，不得反向定义它们的业务语义。

## 3. 当前真实架构盘点

## 3.1 当前仿真执行与结果存储链路

当前仿真执行的权威入口是 `domain/services/simulation_service.py` 中的 `SimulationService`。

它当前负责：

- 接收仿真请求
- 根据文件类型选择执行器
- 执行仿真
- 将结果补充版本号与会话信息
- 通过 `SimulationResultRepository` 保存结果到项目目录下的 `.circuit_ai/sim_results/`
- 发布仿真开始、进度、完成、错误事件

结果存储的权威组件是 `domain/simulation/service/simulation_result_repository.py`。

它当前负责：

- 将 `SimulationResult` 持久化为 JSON 文件
- 读取单个结果文件
- 列出最近仿真结果
- 获取最新结果
- 删除已有结果文件

结论：

- 仿真执行与结果存储已经具备稳定的后端权威路径
- 显示层不应重写结果持久化逻辑
- Web 化应建立在现有结果对象和结果文件之上，而不是额外引入一套前端专用结果格式

## 3.2 当前仿真启动入口与执行边界

当前仿真启动入口由 `presentation/simulation_command_controller.py` 统一协调。

当前事实如下：

- 仿真只能针对当前活动编辑器中的可仿真电路文件启动
- 启动前会自动保存当前脏文件
- 控制器只维护运行按钮状态，不承担结果展示职责
- 仿真运行与仿真结果展示已经在架构上分离

这条边界在重构后必须继续保留。

显示层不得再反向触发第二套“运行仿真”语义，也不得拥有自己的运行状态真相。

## 3.3 当前仿真结果显示总协调器

当前显示层总协调器是 `presentation/panels/simulation/simulation_tab.py`。

它当前承担：

- 订阅仿真开始、完成、错误、项目切换、语言切换等事件
- 在项目打开后恢复最近结果
- 在仿真完成后加载结果
- 调用指标面板、图表面板、波形面板、日志面板等子组件
- 协调“添加至对话”入口
- 处理空状态与状态提示显示

但它当前是一个典型的 Qt Widget 聚合器，并且直接持有多个显示子组件实例。

这意味着：

- 当前显示层虽然有中心协调器，但它仍是原生 Widget 结构
- Web 化不能只是给某个子图表换皮，而要把整个显示面板收敛成单一 Web Surface

## 3.4 当前仿真结果面板结构

当前 `SimulationResultTabView` 位于 `presentation/panels/simulation/simulation_tab_widgets.py`，它基于 `QTabWidget` 把显示拆成多个 tab：

- 仿真指标
- 图表
- 波形
- 分析信息
- 原始数据
- 输出日志
- 数据导出

这套结构说明当前功能已经比较完整，也暴露出一个不能忽视的现实约束：当前仿真显示区域的可用空间有限，多 `tab` 不是偶然产物，而是当前信息密度下合理的组织方式。

- `metrics` / `chart` / `waveform` / `analysis_info` / `raw_data` / `log` / `export` 的信息密度和交互模式差异很大，强行塞进单页会直接压缩主内容区
- 真正需要删除的是 `QTabWidget + 多个 Qt widget` 组成的旧显示壳，而不是 tab 作为信息架构本身
- Web 化后的正确方向是保留“多结果 tab”的心智模型，但把它收敛为一个共享状态、共享动作协议、共享视觉语言的 Web tab 工作区

## 3.5 当前波形交互能力与真实边界

当前 `WaveformWidget` 已经实现了较完整的波形交互能力，且相当一部分能力已经从单体 widget 中被拆出：

- `domain/simulation/data/waveform_data_service.py`
  - 作为波形数据访问统一入口
  - 管理金字塔、多分辨率数据、缓存、视口数据读取、表格快照

- `presentation/panels/simulation/waveform_measurement_support.py`
  - 负责测量计算
  - 负责根据双光标计算差值、频率、各信号在光标处的数值

- `presentation/panels/simulation/waveform_viewport_manager.py`
  - 负责视口、域范围、视口重载和并行坐标映射

当前波形功能已支持：

- 多信号叠加
- 电压 / 电流双 Y 轴
- 信号树选择
- 框选放大
- Fit 全视图
- Grid / Legend 控制
- 双光标启停与移动
- 测量栏显示
- 视口变化驱动数据重载
- 波形截图导出
- 波形导出 bundle
- 添加当前波形到对话

这说明：

- 现有波形域能力不是问题核心
- 当前问题主要在于“这些能力被绑定在 Qt + pyqtgraph 的显示壳中”
- 重构的正确方向不是重写这些域能力，而是把它们重新挂接到 Web 前端上

## 3.6 当前分析图表显示能力

当前 `analysis_chart_viewer.py` 已经能根据分析类型生成图表规格：

- `tran` 对应时域波形图
- `ac` 对应 Bode Overlay
- `dc` 对应 DC Sweep
- `noise` 对应噪声谱图

其本质是：

- 已经存在“结果 -> 图表规格”的抽象
- 图表类型选择并不完全依赖界面猜测
- 图表层已经开始脱离单纯 widget 硬编码

这是 Web 化图表层的重要基础。

重构后应进一步把这条路径收敛为：

- 结果对象与图表规格是权威
- Web 前端图表组件只消费规格，不再自创分析语义

## 3.7 当前添加至对话的链路

当前“添加到对话”由 `simulation_conversation_attachment_coordinator.py` 协调。

它当前依赖：

- 指标导出工件
- 输出日志导出工件
- 当前图表截图
- 当前波形截图

并通过事件总线把文件附件发回对话面板。

这说明：

- “添加至对话”并不是显示层的本地假动作
- 它本质上是工件导出和会话附件联动流程
- Web 化后仍然必须保留后端工件导出权威路径

### 3.7.1 当前原始数据、日志与导出交互

当前仿真细节面并不是被动说明区，而是多个有独立交互语义的功能面：

- `AnalysisInfoPanel`
  - 展示分析类型、执行器、坐标轴信息、参数信息

- `RawDataTable`
  - 基于 `WaveformDataService.build_table_snapshot()` 展示结果快照
  - 支持跳转到指定行
  - 支持按 X 值跳转
  - 支持按列数值搜索

- `OutputLogViewer`
  - 支持关键词搜索高亮
  - 支持日志级别过滤
  - 支持跳转到首个错误
  - 支持刷新与添加至对话

- `SimulationExportPanel`
  - 支持勾选导出类别
  - 支持手动导出到任意目录
  - 支持项目自动导出

结论：

- 这些功能不能在 React 重构时被统称为“右侧详情区”后再弱化处理
- 它们应继续作为一等功能面拥有明确的独立 tab 策略

### 3.7.2 当前历史结果与 OP 结果入口

当前仍然存在两个边缘但真实的结果展示入口：

- `SimulationMetricsSummaryPanel` 的 history 按钮会打开迭代历史记录
- `SimulationTab` 的空状态按钮可尝试恢复最近结果
- `op` 分析结果会在加载后通过 `OPResultDialog` 单独弹窗显示

这说明：

- 历史结果浏览不是临时调试能力，而是用户真实会使用的结果回看入口
- 工作点结果展示不是普通图表/波形的退化情况，而是独立结果类型
- React 重构必须为两者提供明确迁移策略，而不能继续依赖旧 Qt 对话框

### 3.7.3 当前旧设计与死代码信号

当前仿真显示代码里已经出现若干明显的旧设计或半残留实现信号：

- `tuning_panel.py` 及其在 `presentation.panels.simulation.__init__` 中的导出仍然存在，但主仿真结果面板和 `SimulationResultTabView` 已无接入
- `SimulationViewModel.request_simulation()` 与 `presentation/simulation_command_controller.py` 的启动职责重复，显示层不应继续持有第二套运行入口
- `ChartViewer` 维护 `_pages` 列表，但 `_current_page()` 始终只返回第一页，说明旧的多页面抽象已经退化为半残留结构
- `SimulationStatusBanner` 持有 progress bar，而当前显示路径只使用运行中 banner，没有形成完整的进度消费闭环

这些都不应在 Web 重构时原样迁移，而应在阶段计划中明确判定为“要接入的新权威路径”或“要物理删除的旧残留”。

## 3.8 当前撤回与仿真显示的关系

当前撤回机制已经通过 `ConversationRollbackService` 和 `EVENT_WORKSPACE_SYNC_REQUIRED` 与工作区刷新做了明确分离。

对于仿真显示而言，需要明确以下事实：

- 撤回不是仿真面板的内部行为
- 撤回会影响工作区文件与可能的当前结果有效性
- 仿真显示层应该响应新的结果状态和工作区同步，而不是把撤回逻辑塞进仿真前端

这意味着重构后必须保持：

- 撤回是独立后端权威链路
- 仿真前端只消费结果与工作区同步后的状态变化
- 不得在仿真 Web 前端中生成第二套 rollback 逻辑

## 3.9 当前项目中的 Web 承载能力

当前项目已经具备成熟的 Web 承载基础：

- `presentation/core/web_resource_host.py` 已建立 `cai://app` 资源宿主
- 工作区标签栏、文件浏览器、部分文档查看器已经使用 `QWebEngineView`
- 资源加载方式和 Web 宿主模式在项目内已被验证

这说明：

- 仿真显示 Web 化不是架构方向突变
- 当前工程已经具备统一的 Web 资源分发能力
- 正确方向应是复用现有 `cai://app` 与 `QWebChannel` 模式，而不是另外搭一套临时前端加载方式

## 4. 核心判断

### 4.1 本轮重构的本质不是“把图画到浏览器里”

真正要重构的是：

- 显示壳层
- 状态收敛方式
- 交互路由方式
- 面板组织方式
- 多显示子系统之间的统一性

如果只是把某个图从 pyqtgraph 改成前端图表库，而保留 Qt tab、Qt 面板、Qt 子组件协调逻辑不动，那么本轮不会得到统一的架构，只会得到一套更复杂的混合系统。

### 4.2 必须建立单一 Web 仿真显示入口

仿真显示层必须最终收敛为：

- 一个原生 Qt 宿主面板
- 一个 Web 仿真结果 Surface
- 一个桥接协议
- 一个统一前端状态模型

不能长期保留：

- 旧 `SimulationTab` + 多个 Qt 子页面
- 新 Web 波形图
- 旧 Qt 图表页
- 旧 Qt 导出页
- 旧 Qt 状态栏

这样的双轨结构。

### 4.3 显示层必须只消费结果，不得定义业务

Web 化之后必须继续坚持：

- 仿真执行权威属于执行链路
- 结果存储权威属于结果仓库
- 回滚权威属于对话撤回服务
- 测量计算权威属于测量支持层
- 视口数据重载权威属于波形数据与视口管理层

前端只负责：

- 组织显示结构
- 派发用户意图
- 渲染权威状态
- 持有局部瞬时 UI 状态

### 4.4 视觉重构必须伴随结构重构

你的要求不是“小修小改的实现效果”，因此本轮必须同步解决这些结构问题：

- 同一结果在多个 Qt 子页面中被拆散显示
- 波形、分析图、导出、分析信息等显示系统缺少统一布局体系
- 同一个页面中的自适应、响应式、工具栏、信息面板没有统一设计语言
- 多个显示组件各自管理状态，而不是统一消费一份前端状态

### 4.5 不允许伪 Web 化

以下做法必须明确禁止：

- 在 Qt 页面里嵌一个单独 Web 图，但其它部分继续保留原生旧布局
- 用 Web 只包一层皮肤，而保留原来的 Widget 树为真实显示体系
- 让前端自己重新解析结果 JSON 并定义新语义
- 前端和 Python 同时各存一份当前结果状态
- 为兼容旧页面长期保留旧 tab 体系

### 4.6 必须保留多 tab 心智模型，但删除 Qt tab 壳层

必须明确区分两件事：

- 需要删除的是 `QTabWidget` 与多个 Qt 子 widget 组成的旧壳层
- 需要保留的是多 `tab` 作为结果组织方式本身

因此 Web 重构后的前端应继续提供清晰的顶层结果 tab，例如：

- 仿真指标
- 图表
- 波形
- 分析信息
- 原始数据
- 输出日志
- 数据导出
- 按结果类型追加的专门 tab（如工作点结果）

窄宽度下也不应把所有 tab 内容强行堆成单页，而应保持“单个活动 tab + 可滚动或折叠的 tab 导航”的结构。

## 5. 目标架构

## 5.1 总体目标结构

 重构后的结构应为：

 - `SimulationPanel` 或现有 `SimulationTab` 退化为原生 Qt 宿主入口
 - 新增单一 `SimulationWebHost`
 - 新增单一 `SimulationWebBridge`
 - 新增单一 `SimulationFrontendStateSerializer`
 - 新增单一 React/TypeScript 仿真前端工程
 - Web 前端内部采用“顶层 tab 直接作为主界面”的结构：
   - 最上层直接是结果 tab 条，不额外保留“仿真结果”标题栏
   - 不设置共享顶部摘要区
   - 顶层结果 tabs 至少包括：指标、图表、波形、分析信息、原始数据、输出日志、导出、历史结果、工作点结果*
   - 运行中、空状态、错误、确认态只作为当前 tab 的局部状态层处理

## 5.2 前后端职责划分

### 后端继续保留的职责

- 仿真执行和运行状态发布
- 仿真结果仓库存取
- 仿真结果对象构建
- 波形多分辨率数据服务
- 视口数据重载
- 测量结果计算
- 图表规格生成
- 导出工件生成
- 添加至对话的附件桥接
- 项目切换与工作区同步

### 前端承担的职责

- 顶层 tab 导航与活动 tab 状态维护
- 维护 tab-first 的主界面结构，不额外生成摘要头
- 各 tab 的局部工具栏和上下文动作区
- 波形画布与分析图表画布呈现
- 信号选择、光标工具、框选反馈、测量栏显示
- 原始数据的跳转、搜索与快照浏览体验
- 输出日志的搜索、过滤、错误跳转体验
- 历史结果 tab 与当前结果切换体验
- 工作点结果 tab 的专门展示体验
- 导出 tab 内的手动导出选择体验
- 基于后端状态的声明式渲染

## 5.3 推荐的 Web 技术栈

推荐采用：

- React 18
- TypeScript
- Vite 作为构建工具
- 基于 `QWebChannel` 的桥接协议
- 通过 `cai://app` 提供本地资源
- 基于 Canvas 或 WebGL 的高密度图形渲染方案
- 统一的设计 token 与 CSS 体系

不建议采用：

- 重型服务端渲染框架
- 需要网络依赖的在线资源
- 在前端引入第二套业务状态管理真相
- 让图表库直接决定业务结构

## 5.4 前端状态模型目标

前端必须只消费一份结构化状态，不允许从多个后端对象零散拼装 UI。

建议状态至少分为以下层次：

- `simulation_runtime`
  - 当前是否空闲、运行中、错误、等待确认
  - 当前项目路径

## 6. 页面与交互结构重构目标

## 6.1 页面总体布局

不应继续保留“QTabWidget 驱动的 Qt 仿真结果页面”，但必须保留多 `tab` 信息架构。

要删除的是 Qt Widget 的 tab 壳层，而不是 tab 作为结果组织方式本身。

建议重构后的 Web 页面采用“最上层直接 tab 条 + 单活动 tab 内容区”的结构：

- 最上方直接是结果 tab 条，不额外增加“仿真结果”标题行
- 不设置共享顶部摘要区
- 主内容区始终只渲染当前活动 tab
- 每个 tab 内部按自己的交互密度组织局部分栏、工具栏和状态栏
- 窄宽度下保持单个活动 tab，使用横向滚动、overflow 菜单或下拉切换，而不是把全部结果堆成单页

顶层 tab 至少应覆盖：

- 指标
- 图表
- 波形
- 分析信息
- 原始数据
- 输出日志
- 导出
- 历史结果
- 工作点结果*

其中：

- `历史结果` 必须作为同级 tab 出现，而不是独立页面、弹层或抽屉
- `工作点结果*` 只在 `op` 结果存在时出现
- 主界面中不再单独显示 completed / analysis / timestamp / executor / score / refresh / auto-export 状态等摘要信息
- 如果某类信息确有保留价值，也只能回到所属 tab 内部，不得重新抽出一条全局摘要栏

## 6.2 波形视图目标结构

波形 tab 内部建议拆为三层：

- `WaveformToolbar`
  - Grid
  - Legend
  - Cursor A / B
  - Fit
  - Add to Conversation
  - 未来可扩展导出、截图、显示密度等动作

- `WaveformCanvasSurface`
  - 负责真正的波形绘制
  - 接收结构化轨迹数据与视口数据
  - 显示框选、光标、图例、hover 信息

- `WaveformMeasurementPanel`
  - 展示 A/B 光标位置
  - ΔX / ΔY / slope / frequency
  - 各信号在光标点处的值

信号树应改为波形 tab 内部的 Web 化可折叠信号浏览器，而不是整个仿真页面的常驻全局导航区，支持：

- 分类
- 勾选开关
- 全清除
- 搜索筛选
- 未来支持收藏信号和最近使用信号

## 6.3 分析图表视图目标结构

分析图应保留为独立 chart tab，而不是被压缩成波形 tab 的附属区域，也不应被塞回单页工作台中的共享主画布。

建议其与波形视图共享以下设计语义：

- 统一工具栏风格
- 统一图例风格
- 统一空状态与错误状态
- 统一导出入口
- 统一添加至对话入口
- 统一响应式布局

分析图表组件本身只消费后端给出的图表规格，不再自行决定分析类型如何展示。

## 6.4 详细信息区目标结构

当前分析信息、原始数据、输出日志、导出面板是分散的 Widget。

但在当前面板空间约束下，它们更适合作为独立结果 tab，而不是被强行合并成常驻右侧或底部详情区。

重构后应改为以下独立 Web 结果 tabs：

- `AnalysisInfoTab`
- `RawDataTab`
- `OutputLogTab`
- `ExportTab`

每个 tab 保留自己的局部工具栏和交互模型，例如：

- `RawDataTab` 保留跳转、按值搜索、快照浏览
- `OutputLogTab` 保留过滤、搜索、错误跳转
- `ExportTab` 保留导出项勾选、导出反馈

窄宽度下也不应把这些 tab 内容降级成一条纵向长页面，而应保持“只显示当前活动 tab”的结构。

## 6.5 历史结果与工作点结果目标结构

历史结果和工作点结果必须在本轮文档中拥有明确迁移策略：

- 历史结果不应继续停留在 `IterationHistoryDialog`
- 应迁移为与其他结果面完全同级的 `历史结果` tab
- 选择历史项后，应把对应结果加载回当前这套 tab 工作区，而不是跳回旧 Qt 路径

- `op` 分析结果不应继续依赖 `OPResultDialog`
- 应迁移为与其他结果面同级的 `工作点结果` tab

## 6.6 添加至对话的显示目标

“添加至对话”不应再被设计成顶部共享动作区。

它必须按 tab 内聚，并且只出现在以下结果面中：

- 指标 tab：添加指标 / 摘要到对话
- 图表 tab：添加当前图表到对话
- 波形 tab：添加当前波形到对话
- 输出日志 tab：添加输出日志到对话
- 工作点结果 tab：添加工作点结果到对话

以下 tab 默认不提供“添加至对话”入口：

- 分析信息
- 原始数据
- 导出
- 历史结果

前端只负责展示动作和结果反馈；真正的工件导出与事件发布仍由后端权威协调器完成。

## 7. 必须保留并升级的交互能力

## 7.1 自适应布局

必须支持：

- 顶层结果 tab 条始终保持在页面最上方
- 不存在额外的摘要头或无业务价值的顶部信息栏
- 宽屏下当前活动 tab 内部可使用局部分栏布局
- 中等宽度下波形 tab 的信号区折叠或抽屉化
- 窄宽度下保持单个活动 tab，tab 条允许横向滚动或折叠菜单切换
- 工具栏自动换行但不破坏主动作层级
- 大表格与日志区域各自独立滚动

## 7.2 框选放大与视口重载

必须保留当前的核心语义：

- 用户框选主图区域
- 形成新的视口范围
- 根据视口范围向后端请求更适合该范围的数据密度
- 更新画布而不改变结果对象本身
- 支持 Fit 返回全局视图

这条交互语义必须由前后端共同维护：

- 前端负责交互与视图反馈
- 后端负责数据重采样与视口数据提供

## 7.3 信号选择

必须保留并增强：

- 勾选信号显示/隐藏
- 多信号叠加
- 电压 / 电流双轴区分
- 清除全部信号
- 信号树和当前显示轨迹的双向同步

建议新增统一设计要求：

- 信号项必须显示颜色标识
- 当前高亮信号必须与图中轨迹一致
- 选择体验应支持快速筛选和多选

## 7.4 光标与测量

必须完整保留：

- 光标 A / B 添加与删除
- 光标位置拖动
- A/B 差值
- 时间域频率换算
- 各信号在光标位置的数值显示

但必须进行结构收敛：

- 不再让光标 UI 状态散落在工具按钮、图层和测量栏中各自维护
- 前端应有统一的 cursor state
- 后端测量计算只负责值，不负责 UI 结构

## 7.5 图表与波形导出、添加至对话

必须保留：

- 当前图表截图
- 当前波形截图
- 指标数据导出
- 原始数据导出
- 输出日志导出
- 添加至对话

但必须统一动作语义，同时保持入口按 tab 内聚，避免：

- 每个旧 widget 一个独立导出区域
- 把所有“添加到对话”集中到页面公共工具栏
- 每个旧 widget 一个独立“添加到对话”按钮语义

## 7.6 原始数据交互

必须保留：

- 跳转到指定行
- 按 X 值跳转
- 按列数值搜索
- 快照绑定后的稳定表格浏览
- 当前选择与行数状态反馈

React 重构后的 `RawDataTab` 不得退化成一个只会展示静态表格的只读页面。

## 7.7 输出日志交互

必须保留：

- 关键词搜索与高亮
- 日志级别过滤
- 跳转到首个错误
- 添加输出日志到对话

如果未来仍保留“重新读取日志”能力，它也只能作为输出日志 tab 内的局部动作，而不能重新上浮为页面级刷新按钮。

日志 tab 必须继续被视为一等交互面，而不是附属文本框。

## 7.8 历史结果与工作点结果

必须保留并重构：

- 历史结果作为同级 tab 出现在顶层 tab 条中
- 在历史结果 tab 中选择历史记录后加载该结果进入当前 Web tab 工作区
- 工作点结果的专门展示视图
- 旧 Qt 历史/工作点对话框迁移后的删除闭环

## 7.9 导出与项目自动导出

必须保留：

- 手动勾选导出类别
- 指标、图表、波形、分析信息、原始数据、输出日志的完整导出覆盖
- 项目自动导出
- 添加至对话时复用后端权威工件路径

但必须明确：

- 项目自动导出可以继续作为后端能力存在
- 不要求在主界面顶部常驻显示“自动导出状态 / 最近导出目录”的信息
- 如果确需可视化导出反馈，也只能放在 `导出` tab 内，而不是回到全局摘要区

以下旧设计在重构完成后必须物理删除，而不是弱化保留。

必须删除：

- 基于 `QTabWidget` 的 Qt 仿真结果主页面壳层
- `SimulationResultTabView` 作为长期权威 Qt 容器的职责
- 多个独立 Qt 结果页并行存在的页面组织方式

## 8.2 旧波形显示壳

必须删除：

- `WaveformWidget` 作为最终权威显示壳的地位
- pyqtgraph 驱动的最终交互页面
- Qt 工具栏、Qt 信号树、Qt 测量栏这套完整 UI 壳层

可以保留的仅应是其中已经拆出的、与 Qt 无直接耦合的支持能力，例如数据服务、测量计算、视口管理等。

## 8.3 旧图表显示壳

必须删除：

- Qt 图表页面作为最终权威视觉实现
- 依赖 Qt Widget 分页来组织结果观察维度的方式
- 图表、波形、日志、原始数据各自为政的视觉体系

## 8.4 旧状态分发方式

必须删除：

- 由多个 Qt 子组件各自持有局部业务状态的方式
- 组件各自直接从事件总线读取状态并自作解释的路径
- 结果显示结构由多个 widget 自己拼接的方式

最终必须收敛成：

- 后端统一序列化状态
- 前端统一消费状态
- 动作通过单一 bridge 回流

## 8.5 旧导出和添加至对话入口的散乱设计

必须删除：

- 指标、图表、波形、日志各自独立一套不统一的动作入口设计
- 仅在某个旧面板内部才能触发的导出逻辑
- UI 组件直接决定工件目录结构的路径

## 8.6 旧兼容路径

必须删除：

- Web 页面与 Qt 页面并行维护同一结果显示的兼容逻辑
- 旧 widget 与新 Web surface 同时订阅同一批显示状态
- 仅为兼容旧显示组件而存在的专用中间适配层
- 前端和 Python 双写同一显示状态的过渡分支

## 8.7 脱离主链与半残留实现

必须删除或完成明确归并：

- `tuning_panel.py` 及其在 `presentation.panels.simulation.__init__` 中的导出，如果本轮没有明确的 Web 调参方案
- `SimulationViewModel.request_simulation()` 这类与 `SimulationCommandController` 重复的显示层运行入口
- `ChartViewer` 中 `_pages` / `_current_page()` 这类只剩第一页语义的半残留多页面抽象
- 旧 `IterationHistoryDialog` 与 `OPResultDialog`，在 Web 历史结果 tab 和工作点结果 tab 落地后必须删除
- 显示层中未被真实消费的进度/状态字段与对应 UI 分支，如果它们没有进入新的状态契约

## Phase 3：迁移波形视图与高密度交互

### 当前代码审计结论

- 当前 `WaveformWidget` 仍是一个**单体 Qt widget**，内部同时承载：信号树外壳、工具栏外壳、`pyqtgraph` 画布、双光标状态、测量栏显示、视口重载、导出能力与“添加至对话”入口。这种把 UI 壳、交互状态和后端能力混在一个对象里的设计，不能直接延续到 Web 架构。
- 当前真正可复用的权威后端能力主要是：
  - `WaveformDataService`：初始数据、视口数据、信号解析与分组
  - `WaveformViewportManager`：domain 重建、视口裁剪、双 Y 轴范围映射
  - `WaveformMeasurementSupport`：双光标测量值与频率换算
  - `waveform_export_bundle_builder`：波形导出工件构建
- 当前 `SimulationFrontendStateSerializer.waveform_view` 只暴露 `has_waveform`、`signal_count`、`signal_names`、`can_export`、`can_add_to_conversation`，这个状态粒度**远不足以支撑真正的 Web 波形面**。
- 当前前端 `WaveformTab.tsx` 和 fallback host 中的 `waveform` surface 仍只是 Phase 1/2 占位骨架，还没有承载信号筛选、已显示轨迹状态、视口状态、光标状态、双 Y 轴表达和测量结果。
- 当前 bridge 动作虽然已经是 typed intent，但实际执行仍大量直接落到 `WaveformWidget`；Phase 3 不能继续把这个对象当作长期权威接口，只能把它视为临时 backend runtime 载体或待拆分对象。
- 本轮代码审计还暴露出一个具体坏设计：`clear all signals` 不能等价于“清空整个波形结果上下文”。它的正确语义应当只是**清空当前显示轨迹，同时保留当前结果上下文与信号目录**。本阶段必须保持这个语义收敛，不允许回退。

### 目标

将波形显示能力迁移到 Web 前端，并在迁移过程中把当前波形能力从“单体 Qt widget”重构为“后端权威能力层 + Web 波形交互面”的清晰架构，同时保持现有域能力不降级。

### 此阶段必须先完成的结构拆分

- 必须把“波形数据访问 / 视口重载 / 测量计算 / 导出与对话附件”与“Qt 信号树 / Qt 工具栏 / Qt 测量栏”彻底区分开。
- Web 波形面只能消费**权威状态 + typed bridge 动作**，不能继续依赖 `WaveformWidget` 中某个按钮、复选框、树节点或测量栏标签的私有行为。
- 当前 `WaveformWidget` 如果短期内仍保留，只能作为 backend runtime 私有实现，用于承接 `pyqtgraph` 渲染、现有导出或过渡期后端支持；它的 Qt 工具栏、Qt 信号树、Qt 测量栏不得继续主导新架构。
- 必须为波形建立**专门的权威前端状态契约**，而不是继续把 `waveform_view` 维持在“只有名字列表和布尔标志”的低保真占位形态。
- 前端应统一持有**UI 级交互状态**，包括筛选关键字、当前 hover / highlight、框选中的临时区域、光标拖动中的瞬时状态、侧栏折叠状态等；后端只负责已提交的视口、数据密度和测量结果。

### 必须完成

- Web 信号浏览区与快速筛选
- Web 波形画布
- Grid / Legend / Fit
- 框选放大与视口重载
- 信号显隐、清除全部信号、多信号叠加
- 双光标与测量栏
- 光标 A / B 添加删除、位置拖动、A/B 差值、时间域频率换算、光标点数值显示
- 双 Y 轴语义
- 当前波形导出与添加至对话入口
- 波形专用权威状态契约

### 波形权威状态契约至少必须覆盖

- 信号目录：按 `voltage / current / other` 分组后的可浏览信号列表
- 每个信号的稳定标识、显示名、当前颜色、所属轴语义、当前是否显示
- 当前已显示轨迹列表及其显示顺序
- 当前是否启用 `grid` / `legend`
- 当前结果的全局 X 域、当前活动视口 X 域、是否为 log X
- 双 Y 轴的当前语义范围或其前端绘制所需的最小表达
- 光标 A / B 的可见性、已提交位置、差值摘要
- 测量栏摘要，以及各已显示信号在光标位置的数值结果
- 当前波形是否可导出、是否可添加至对话

约束说明：

- 这份状态必须由后端权威序列化给前端，不能由 React 在本地根据零散字段临时拼出第二份业务真相。
- 前端本地可以维护筛选词、hover、临时拖拽框和展开/折叠等**纯 UI 状态**，但不能在本地伪造与后端冲突的波形真相。

### 执行时必须落实的页面细节

- `波形` tab 内部保留自己的局部工具栏，不借用全局页面头部
- “添加至对话”按钮只出现在 `波形` tab 内部工具栏或局部动作区
- 信号树、画布、测量栏都属于 `波形` tab 自己的内部布局，不升级为全局导航结构
- `波形工具栏` 必须做成低高度单行紧凑工具条，目标高度约 `28-32px`
- `测量栏` 必须做成低高度单行信息条，目标高度约 `24-28px`，只默认显示关键测量值
- 垂直空间分配上必须明确“画布优先”，工具栏和测量栏只能占最小必要高度
- 窄宽度下优先折叠信号树和次要控制项，不先牺牲波形画布高度
- 信号项必须显示颜色标识，当前高亮信号必须与图中轨迹一致
- 不把旧 `WaveformWidget` 的 Qt 工具栏 / 信号树 / 测量栏壳整块搬进 Web，只复用后端或无 Qt 耦合的支持能力
- 波形信号项除了名称外，还必须明确显示颜色标识与轴语义标记，避免用户无法分辨左轴/右轴轨迹
- `清除全部信号` 只清空当前显示轨迹，不清空当前结果上下文、不清空信号目录、不退回到“无结果”状态
- 默认测量栏必须保持单行紧凑摘要；如果要显示每个信号在光标处的详细数值，应进入局部可展开细节区或次级浮层，不能回到旧 Qt 两行测量壳
- 窄宽度下优先把信号浏览区做成折叠面、抽屉或局部 overlay，不允许把它升级成全局导航层

### 此阶段必须完整迁移的交互语义

- 用户框选主图区域后，必须形成新的视口范围，并根据该范围向后端请求更适合的数据密度
- `Fit` 必须返回全局视图，而不是仅做前端局部缩放重置
- 波形选择体验必须支持快速筛选、多选、显隐切换、清除全部信号，以及信号树与当前轨迹的双向同步
- 光标 A / B 的添加、删除、拖动、差值计算、频率换算、各信号在光标位置的数值显示必须完整迁移
- 光标状态不得继续散落在工具按钮、图层和测量栏中分别维护，必须收敛为统一前端光标状态
- 前端框选中的临时矩形、光标拖拽中的瞬时位置、hover 高亮等瞬时 UI 状态必须在前端统一管理；只有提交后的视口变更和已提交光标位置才进入后端测量与数据重载流程
- 双 Y 轴语义必须与后端信号分类保持一致：电压默认左轴，电流默认右轴；前端不能重新发明一套与后端分类脱节的轴分配规则
- `Fit` 的语义必须包含“回到全局 X 域 + 重新加载更适合全局视图的数据密度”，而不是只把当前前端 canvas 缩放矩阵重置

### 必须坚持的实现原则

- 视口数据仍由后端权威服务提供
- 前端画布只负责显示与交互
- 测量计算仍由后端支持层产出
- 光标和框选的 UI 状态在前端统一管理
- 导出与“添加至对话”继续复用后端权威工件路径，不在前端单独拼装第二套波形工件
- Bridge 继续表达用户意图，不暴露 Qt 壳私有按钮 / 私有槽函数语义

### 此阶段必须完成的结构收敛

- `SimulationFrontendStateSerializer.waveform_view` 必须从当前的占位字段升级为可支撑完整 Web 波形面的结构化状态；仅保留 `signal_names` + 布尔标志的状态形态不得继续作为长期权威契约
- `frontend/simulation-panel/src/components/tabs/WaveformTab.tsx` 与 fallback host 中当前的占位波形页必须在本阶段结束前退出主路径，不能继续停留在“信号列表 + 显示按钮 + 假测量条”的骨架阶段
- `WaveformWidget` 中的 Qt 工具栏、Qt 信号树、Qt 测量栏如果不再承担主路径职责，就不得继续反向塑造 bridge 协议或前端状态结构
- 未被主路径消费的波形内部 Qt 信号、分散状态桥接和与旧壳强耦合的交互通道，不得继续保留为“也许以后会用”的悬空能力
- 如果短期仍复用 `WaveformWidget` 的 `pyqtgraph` 画布、导出或过渡渲染能力，就必须把这些能力与其 Qt 壳显式拆分；进入后续阶段后，应准备对旧壳做物理删除，而不是长期隐藏保留

## 10. 实现细节要求

## 10.1 状态序列化必须集中管理

不得把仿真前端所需字段散落在：

- `SimulationTab`
- `SimulationViewModel`
- 各子组件
- bridge

之间各写一部分。

必须建立集中序列化层，统一负责：

- 状态字段命名
- 前端 payload 结构
- 结果摘要组装
- 波形视图状态组装
- 详细信息区数据组装

## 10.2 动作桥接只表达用户意图

桥接接口不得直接表达最终 UI 状态。

桥接动作只应该表达：

- 用户要切换什么
- 用户要执行什么
- 用户要请求哪个交互

最终界面状态必须以后端回流的新状态为准。

## 10.3 图形绘制层必须和数据获取层分离

无论最终使用何种前端图形绘制方案，都必须保持：

- 绘制层不负责生成数据
- 绘制层不负责定义波形语义
- 绘制层不负责结果持久化
- 绘制层不负责测量业务计算

## 10.4 添加至对话必须继续走工件路径

前端不得把“添加至对话”退化成把当前页面文本直接塞给对话输入框。

必须继续保留：

- 工件导出
- 文件路径发布
- 对话附件事件流

## 10.5 响应式布局必须是正式设计目标

不能等完成后再用补丁修窄宽度问题。

从一开始就必须明确：

- 结果 tab 条如何滚动、折叠或进入 overflow 菜单
- 波形 tab 内信号树如何折叠/抽屉化
- 当前活动 tab 内局部分栏如何重新排布
- 工具栏如何换行
- 表格和日志如何独立滚动

## 11. 验收标准

当以下条件同时满足时，视为本轮仿真 Web 化视觉重构完成：

- 仿真显示已经由单一 Web Surface 承载
- Web 前端的最上层直接就是结果 tab 条，不存在额外“仿真结果”标题行或共享摘要栏
- Web 前端在有限面板空间内保持清晰的多 tab 结构，而不是把全部内容堆成单页
- 仿真执行、结果存储、撤回机制与显示层边界清晰且未被破坏
- 波形、分析图、分析信息、原始数据、日志、导出、添加至对话能力全部保留
- 历史结果作为同级 tab、工作点结果作为条件性 tab、原始数据搜索跳转、日志过滤跳转、项目自动导出能力全部保留
- “添加至对话”只出现在指标、图表、波形、输出日志、工作点结果这些指定 tab 中
- 框选放大、Fit、光标、测量、信号选择、双轴显示、自适应布局全部正确
- 前端只消费一份统一状态，不存在多套显示真相
- bridge 动作只表达用户意图，不直接操纵 UI DOM 结果
- 旧 Qt 显示壳、旧 tab 结构、旧兼容路径已经被物理删除
- 已确认的死代码、重复入口、半残留抽象已经被物理删除，而不是继续悬挂在仓库中
- 仓库中不存在“新 Web 前端 + 旧 Qt 页面长期并行”的保留结构

## 12. 最终判断

这次重构的正确目标不是“把一个仿真图换成网页图”，而是建立：

- 后端继续保持仿真与结果权威
- 前端用单一 Web 技术栈在有限面板空间内承载 tab-first 的多结果工作区
- 两者通过单一桥接协议协作
- 旧 Qt 显示体系在 cutover 后被彻底删除

只有这样，最终得到的才会是：

- 视觉统一
- 行为稳定
- 架构清晰
- 可扩展
- 对 AI 和人类开发者都可维护的仿真显示系统
