# 仿真领域 Web 化视觉重构设计文档

## 1. 文档目标

本文档用于指导 `circuit_design_ai` 的仿真领域视觉显示重构。

本轮重构不是对现有仿真功能做局部美化，也不是继续在 Qt Widgets 上追加样式补丁，而是要建立一套新的、统一的、权威的 Web 化仿真显示架构，使仿真领域在保持现有核心能力正确的前提下，完成以下目标：

- 将仿真结果显示、波形交互、分析信息与导出展示统一迁移到 Web 技术栈
- 保持仿真执行、仿真结果存储、撤回机制三者与显示层彻底分离
- 保留并升级当前已经存在的能力，包括图表绘制、信号选择、框选放大、光标、测量、添加至对话、自适应布局等
- 建立单一权威的前端显示实现，避免 Qt Widget 页面与 Web 页面长期双轨并存
- 在重构完成后，从代码层面彻底删除旧显示设计、旧入口、旧兼容路径

本文档不写任何具体代码，只写：

- 当前真实架构事实
- 重构边界与核心判断
- 目标架构
- 分阶段执行步骤
- 关键实现细节
- 必须彻底根除的旧设计
- 验收标准与闭环检查项

---

## 2. 重构边界与非目标

### 2.1 本轮重构边界

本轮聚焦于仿真领域的显示与交互表现层，范围包括：

- 仿真结果主面板的整体布局
- 状态提示与空状态显示
- 指标摘要显示
- 分析图表显示
- 波形图显示
- 分析信息、原始数据、输出日志、导出信息等结果面板
- 波形交互能力，包括信号选择、框选放大、视口重载、光标、测量、图例、网格、适配大小
- 添加到对话的显示入口与交互入口
- Web 技术栈下的资源加载、桥接协议与状态下发方式

### 2.2 明确不做的事

本轮不应演变为以下方向：

- 不重写仿真求解器
- 不重写仿真执行器注册与执行链路
- 不改变仿真结果文件的持久化语义
- 不把撤回机制改成仿真面板内部逻辑
- 不让前端成为仿真结果、撤回、导出或对话附件的第二权威源
- 不把整个应用外壳改成 Web
- 不为了过渡长期保留 Qt 版仿真结果界面和 Web 版界面同时存在

### 2.3 必须保留的后端权威边界

以下能力在本轮之后仍必须由后端权威维护：

- 仿真执行入口与运行生命周期
- 仿真结果文件落盘与读取
- 仿真结果对象与其元数据
- 回滚与工作区同步机制
- 波形数据多分辨率访问与缓存
- 测量计算逻辑
- 添加到对话所需的工件导出

显示层只允许消费这些权威结果，不得反向定义它们的业务语义。

## 3. 当前真实架构盘点

## 3.1 当前仿真执行与结果存储链路

当前仿真执行的权威入口是 `domain/services/simulation_service.py` 中的 `SimulationService`。

它当前负责：

- 接收仿真请求
- 根据文件类型选择执行器
- 执行仿真
- 将结果补充版本号与会话信息
- 通过 `SimulationResultRepository` 保存结果到项目目录下的 `.circuit_ai/sim_results/`
- 发布仿真开始、进度、完成、错误事件

结果存储的权威组件是 `domain/simulation/service/simulation_result_repository.py`。

它当前负责：

- 将 `SimulationResult` 持久化为 JSON 文件
- 读取单个结果文件
- 列出最近仿真结果
- 获取最新结果
- 删除已有结果文件

结论：

- 仿真执行与结果存储已经具备稳定的后端权威路径
- 显示层不应重写结果持久化逻辑
- Web 化应建立在现有结果对象和结果文件之上，而不是额外引入一套前端专用结果格式

## 3.2 当前仿真启动入口与执行边界

当前仿真启动入口由 `presentation/simulation_command_controller.py` 统一协调。

当前事实如下：

- 仿真只能针对当前活动编辑器中的可仿真电路文件启动
- 启动前会自动保存当前脏文件
- 控制器只维护运行按钮状态，不承担结果展示职责
- 仿真运行与仿真结果展示已经在架构上分离

这条边界在重构后必须继续保留。

显示层不得再反向触发第二套“运行仿真”语义，也不得拥有自己的运行状态真相。

## 3.3 当前仿真结果显示总协调器

当前显示层总协调器是 `presentation/panels/simulation/simulation_tab.py`。

它当前承担：

- 订阅仿真开始、完成、错误、项目切换、语言切换等事件
- 在项目打开后恢复最近结果
- 在仿真完成后加载结果
- 调用指标面板、图表面板、波形面板、日志面板等子组件
- 协调“添加至对话”入口
- 处理空状态与状态提示显示

但它当前是一个典型的 Qt Widget 聚合器，并且直接持有多个显示子组件实例。

这意味着：

- 当前显示层虽然有中心协调器，但它仍是原生 Widget 结构
- Web 化不能只是给某个子图表换皮，而要把整个显示面板收敛成单一 Web Surface

## 3.4 当前仿真结果面板结构

当前 `SimulationResultTabView` 位于 `presentation/panels/simulation/simulation_tab_widgets.py`，它基于 `QTabWidget` 把显示拆成多个 tab：

- 仿真指标
- 图表
- 波形
- 分析信息
- 原始数据
- 输出日志
- 数据导出

这套结构说明当前功能已经比较完整，但也暴露出一个根本问题：

- 当前页面结构是“按 Widget/Tab 拆分”的，而不是“按统一结果视图状态拆分”的
- 结果的多个观察维度被分割为多个 Qt tab，各组件各自维护局部显示语义
- 这不利于 Web 化后的统一布局、自适应能力和信息联动

## 3.5 当前波形交互能力与真实边界

当前 `WaveformWidget` 已经实现了较完整的波形交互能力，且相当一部分能力已经从单体 widget 中被拆出：

- `domain/simulation/data/waveform_data_service.py`
  - 作为波形数据访问统一入口
  - 管理金字塔、多分辨率数据、缓存、视口数据读取、表格快照

- `presentation/panels/simulation/waveform_measurement_support.py`
  - 负责测量计算
  - 负责根据双光标计算差值、频率、各信号在光标处的数值

- `presentation/panels/simulation/waveform_viewport_manager.py`
  - 负责视口、域范围、视口重载和并行坐标映射

当前波形功能已支持：

- 多信号叠加
- 电压 / 电流双 Y 轴
- 信号树选择
- 框选放大
- Fit 全视图
- Grid / Legend 控制
- 双光标启停与移动
- 测量栏显示
- 视口变化驱动数据重载
- 波形截图导出
- 波形导出 bundle
- 添加当前波形到对话

这说明：

- 现有波形域能力不是问题核心
- 当前问题主要在于“这些能力被绑定在 Qt + pyqtgraph 的显示壳中”
- 重构的正确方向不是重写这些域能力，而是把它们重新挂接到 Web 前端上

## 3.6 当前分析图表显示能力

当前 `analysis_chart_viewer.py` 已经能根据分析类型生成图表规格：

- `tran` 对应时域波形图
- `ac` 对应 Bode Overlay
- `dc` 对应 DC Sweep
- `noise` 对应噪声谱图

其本质是：

- 已经存在“结果 -> 图表规格”的抽象
- 图表类型选择并不完全依赖界面猜测
- 图表层已经开始脱离单纯 widget 硬编码

这是 Web 化图表层的重要基础。

重构后应进一步把这条路径收敛为：

- 结果对象与图表规格是权威
- Web 前端图表组件只消费规格，不再自创分析语义

## 3.7 当前添加至对话的链路

当前“添加到对话”由 `simulation_conversation_attachment_coordinator.py` 协调。

它当前依赖：

- 指标导出工件
- 输出日志导出工件
- 当前图表截图
- 当前波形截图

并通过事件总线把文件附件发回对话面板。

这说明：

- “添加至对话”并不是显示层的本地假动作
- 它本质上是工件导出和会话附件联动流程
- Web 化后仍然必须保留后端工件导出权威路径

## 3.8 当前撤回与仿真显示的关系

当前撤回机制已经通过 `ConversationRollbackService` 和 `EVENT_WORKSPACE_SYNC_REQUIRED` 与工作区刷新做了明确分离。

对于仿真显示而言，需要明确以下事实：

- 撤回不是仿真面板的内部行为
- 撤回会影响工作区文件与可能的当前结果有效性
- 仿真显示层应该响应新的结果状态和工作区同步，而不是把撤回逻辑塞进仿真前端

这意味着重构后必须保持：

- 撤回是独立后端权威链路
- 仿真前端只消费结果与工作区同步后的状态变化
- 不得在仿真 Web 前端中生成第二套 rollback 逻辑

## 3.9 当前项目中的 Web 承载能力

当前项目已经具备成熟的 Web 承载基础：

- `presentation/core/web_resource_host.py` 已建立 `cai://app` 资源宿主
- 工作区标签栏、文件浏览器、部分文档查看器已经使用 `QWebEngineView`
- 资源加载方式和 Web 宿主模式在项目内已被验证

这说明：

- 仿真显示 Web 化不是架构方向突变
- 当前工程已经具备统一的 Web 资源分发能力
- 正确方向应是复用现有 `cai://app` 与 `QWebChannel` 模式，而不是另外搭一套临时前端加载方式

## 4. 核心判断

### 4.1 本轮重构的本质不是“把图画到浏览器里”

真正要重构的是：

- 显示壳层
- 状态收敛方式
- 交互路由方式
- 面板组织方式
- 多显示子系统之间的统一性

如果只是把某个图从 pyqtgraph 改成前端图表库，而保留 Qt tab、Qt 面板、Qt 子组件协调逻辑不动，那么本轮不会得到统一的架构，只会得到一套更复杂的混合系统。

### 4.2 必须建立单一 Web 仿真显示入口

仿真显示层必须最终收敛为：

- 一个原生 Qt 宿主面板
- 一个 Web 仿真结果 Surface
- 一个桥接协议
- 一个统一前端状态模型

不能长期保留：

- 旧 `SimulationTab` + 多个 Qt 子页面
- 新 Web 波形图
- 旧 Qt 图表页
- 旧 Qt 导出页
- 旧 Qt 状态栏

这样的双轨结构。

### 4.3 显示层必须只消费结果，不得定义业务

Web 化之后必须继续坚持：

- 仿真执行权威属于执行链路
- 结果存储权威属于结果仓库
- 回滚权威属于对话撤回服务
- 测量计算权威属于测量支持层
- 视口数据重载权威属于波形数据与视口管理层

前端只负责：

- 组织显示结构
- 派发用户意图
- 渲染权威状态
- 持有局部瞬时 UI 状态

### 4.4 视觉重构必须伴随结构重构

你的要求不是“小修小改的实现效果”，因此本轮必须同步解决这些结构问题：

- 同一结果在多个 Qt 子页面中被拆散显示
- 波形、分析图、导出、分析信息等显示系统缺少统一布局体系
- 同一个页面中的自适应、响应式、工具栏、信息面板没有统一设计语言
- 多个显示组件各自管理状态，而不是统一消费一份前端状态

### 4.5 不允许伪 Web 化

以下做法必须明确禁止：

- 在 Qt 页面里嵌一个单独 Web 图，但其它部分继续保留原生旧布局
- 用 Web 只包一层皮肤，而保留原来的 Widget 树为真实显示体系
- 让前端自己重新解析结果 JSON 并定义新语义
- 前端和 Python 同时各存一份当前结果状态
- 为兼容旧页面长期保留旧 tab 体系

## 5. 目标架构

## 5.1 总体目标结构

重构后的结构应为：

- `SimulationPanel` 或现有 `SimulationTab` 退化为原生 Qt 宿主入口
- 新增单一 `SimulationWebHost`
- 新增单一 `SimulationWebBridge`
- 新增单一 `SimulationFrontendStateSerializer`
- 新增单一 React/TypeScript 仿真前端工程
- Web 前端内部统一承载以下内容：
  - 结果概览
  - 指标摘要
  - 波形视图
  - 分析图表视图
  - 分析信息
  - 原始数据
  - 输出日志
  - 导出与添加至对话入口
  - 历史结果入口
  - 运行状态与空状态

## 5.2 前后端职责划分

### 后端继续保留的职责

- 仿真执行和运行状态发布
- 仿真结果仓库存取
- 仿真结果对象构建
- 波形多分辨率数据服务
- 视口数据重载
- 测量结果计算
- 图表规格生成
- 导出工件生成
- 添加至对话的附件桥接
- 项目切换与工作区同步

### 前端承担的职责

- 统一页面布局
- 统一视觉语言
- 面板切换与响应式组织
- 统一工具栏和上下文操作区
- 波形画布与分析图表画布呈现
- 信号选择、光标工具、框选反馈、测量栏显示
- 原始数据、分析信息、日志、导出概览的浏览体验
- 基于后端状态的声明式渲染

## 5.3 推荐的 Web 技术栈

推荐采用：

- React 18
- TypeScript
- Vite 作为构建工具
- 基于 `QWebChannel` 的桥接协议
- 通过 `cai://app` 提供本地资源
- 基于 Canvas 或 WebGL 的高密度图形渲染方案
- 统一的设计 token 与 CSS 体系

不建议采用：

- 重型服务端渲染框架
- 需要网络依赖的在线资源
- 在前端引入第二套业务状态管理真相
- 让图表库直接决定业务结构

## 5.4 前端状态模型目标

前端必须只消费一份结构化状态，不允许从多个后端对象零散拼装 UI。

建议状态至少分为以下层次：

- `simulation_runtime`
  - 当前是否空闲、运行中、完成、错误
  - 当前项目路径
  - 当前结果路径
  - 最近加载时间
  - 状态提示信息

- `result_summary`
  - 电路文件信息
  - 分析类型
  - 时间戳
  - 执行器
  - 成功/失败状态
  - 综合评分

- `metrics_view`
  - 指标卡片列表
  - 类别分组
  - 排序与筛选状态

- `waveform_view`
  - 当前展示的信号树
  - 当前选中信号
  - 当前显示的轨迹列表
  - 光标状态
  - 测量状态
  - 网格、图例、Fit、缩放状态
  - 视口范围

- `analysis_chart_view`
  - 图表规格列表
  - 当前图表类型
  - 当前系列开关

- `detail_panels`
  - 分析信息
  - 原始数据
  - 输出日志
  - 导出摘要

- `conversation_attachment_actions`
  - 可添加内容类型
  - 相关按钮状态
  - 导出进度或错误信息

其中，前端允许自己维护的仅限于局部瞬时显示状态，例如：

- 当前展开的侧边栏 section
- 当前 tab / segment 选中项
- 表格滚动位置
- 光标浮层显示状态
- 框选拖拽中的临时 overlay
- 原始数据表的本地排序和筛选条件

## 6. 页面与交互结构重构目标

## 6.1 页面总体布局

不应继续保留“QTabWidget 驱动的仿真结果页面”。

建议重构后的 Web 页面采用统一结果工作台布局：

- 顶部状态与结果总览区
- 左侧可折叠信号与结果导航区
- 中央主可视化区
- 右侧上下文信息区或底部详细信息区
- 移动或窄宽度下自动折叠为垂直堆叠布局

主可视化区应允许在以下视图间统一切换，而不是切到不同 Qt tab：

- 波形主视图
- 分析图主视图
- 表格主视图
- 日志主视图

## 6.2 波形视图目标结构

波形视图建议拆为三层：

- `WaveformToolbar`
  - Grid
  - Legend
  - Cursor A / B
  - Fit
  - Add to Conversation
  - 未来可扩展导出、截图、显示密度等动作

- `WaveformCanvasSurface`
  - 负责真正的波形绘制
  - 接收结构化轨迹数据与视口数据
  - 显示框选、光标、图例、hover 信息

- `WaveformMeasurementPanel`
  - 展示 A/B 光标位置
  - ΔX / ΔY / slope / frequency
  - 各信号在光标点处的值

左侧信号树应改为 Web 化的可折叠信号浏览器，支持：

- 分类
- 勾选开关
- 全清除
- 搜索筛选
- 未来支持收藏信号和最近使用信号

## 6.3 分析图表视图目标结构

分析图不应继续作为和波形视图完全割裂的另一套面板。

建议其与波形视图共享以下设计语义：

- 统一工具栏风格
- 统一图例风格
- 统一空状态与错误状态
- 统一导出入口
- 统一添加至对话入口
- 统一响应式布局

分析图表组件本身只消费后端给出的图表规格，不再自行决定分析类型如何展示。

## 6.4 详细信息区目标结构

当前分析信息、原始数据、输出日志、导出面板是分散的 Widget。

重构后应改为统一详情层，建议按以下组织：

- `AnalysisInfoSection`
- `RawDataSection`
- `OutputLogSection`
- `ExportArtifactsSection`

这些 section 既可以作为右侧详情面板，也可以在窄宽度下下沉到底部折叠区域。

## 6.5 添加至对话的显示目标

“添加至对话”不应继续散落在多个旧控件中各自实现。

应统一为一套上下文动作入口，明确支持：

- 添加指标
- 添加当前图表
- 添加当前波形
- 添加输出日志

前端只负责展示动作和结果反馈；真正的工件导出与事件发布仍由后端权威协调器完成。

---

## 7. 必须保留并升级的交互能力

## 7.1 自适应布局

必须支持：

- 宽屏下多栏布局
- 中等宽度下左侧信号区折叠、信息区收缩
- 窄宽度下主图优先，其余信息堆叠显示
- 工具栏自动换行但不破坏主动作层级
- 大表格与日志区域各自独立滚动

## 7.2 框选放大与视口重载

必须保留当前的核心语义：

- 用户框选主图区域
- 形成新的视口范围
- 根据视口范围向后端请求更适合该范围的数据密度
- 更新画布而不改变结果对象本身
- 支持 Fit 返回全局视图

这条交互语义必须由前后端共同维护：

- 前端负责交互与视图反馈
- 后端负责数据重采样与视口数据提供

## 7.3 信号选择

必须保留并增强：

- 勾选信号显示/隐藏
- 多信号叠加
- 电压 / 电流双轴区分
- 清除全部信号
- 信号树和当前显示轨迹的双向同步

建议新增统一设计要求：

- 信号项必须显示颜色标识
- 当前高亮信号必须与图中轨迹一致
- 选择体验应支持快速筛选和多选

## 7.4 光标与测量

必须完整保留：

- 光标 A / B 添加与删除
- 光标位置拖动
- A/B 差值
- 时间域频率换算
- 各信号在光标位置的数值显示

但必须进行结构收敛：

- 不再让光标 UI 状态散落在工具按钮、图层和测量栏中各自维护
- 前端应有统一的 cursor state
- 后端测量计算只负责值，不负责 UI 结构

## 7.5 图表与波形导出、添加至对话

必须保留：

- 当前图表截图
- 当前波形截图
- 指标数据导出
- 原始数据导出
- 输出日志导出
- 添加至对话

但必须统一入口和反馈形式，避免：

- 每个旧 widget 一个独立导出区域
- 每个旧 widget 一个独立“添加到对话”按钮语义

## 8. 必须根除的旧设计

以下旧设计在重构完成后必须物理删除，而不是弱化保留。

## 8.1 旧显示壳层

必须删除：

- 基于 `QTabWidget` 的仿真结果主页面结构
- `SimulationResultTabView` 作为长期权威显示容器的职责
- 多个独立 Qt 结果页并行存在的页面组织方式

## 8.2 旧波形显示壳

必须删除：

- `WaveformWidget` 作为最终权威显示壳的地位
- pyqtgraph 驱动的最终交互页面
- Qt 工具栏、Qt 信号树、Qt 测量栏这套完整 UI 壳层

可以保留的仅应是其中已经拆出的、与 Qt 无直接耦合的支持能力，例如数据服务、测量计算、视口管理等。

## 8.3 旧图表显示壳

必须删除：

- Qt 图表页面作为最终权威视觉实现
- 依赖 Widget 分页来组织结果观察维度的方式
- 图表、波形、日志、原始数据各自为政的视觉体系

## 8.4 旧状态分发方式

必须删除：

- 由多个 Qt 子组件各自持有局部业务状态的方式
- 组件各自直接从事件总线读取状态并自作解释的路径
- 结果显示结构由多个 widget 自己拼接的方式

最终必须收敛成：

- 后端统一序列化状态
- 前端统一消费状态
- 动作通过单一 bridge 回流

## 8.5 旧导出和添加至对话入口的散乱设计

必须删除：

- 指标、图表、波形、日志各自独立一套不统一的动作入口设计
- 仅在某个旧面板内部才能触发的导出逻辑
- UI 组件直接决定工件目录结构的路径

## 8.6 旧兼容路径

必须删除：

- Web 页面与 Qt 页面并行维护同一结果显示的兼容逻辑
- 旧 widget 与新 Web surface 同时订阅同一批显示状态
- 仅为兼容旧显示组件而存在的专用中间适配层
- 前端和 Python 双写同一显示状态的过渡分支

## 9. 分阶段执行步骤

## Phase 0：冻结功能基线与状态契约

### 必须完成

- 盘点当前仿真结果显示的全部能力清单
- 盘点波形、分析图、分析信息、日志、导出、添加至对话的真实入口
- 盘点后端权威对象与其字段边界
- 定义新的前端状态契约草案
- 明确哪些状态属于前端瞬时状态，哪些必须由后端下发

### 阶段产物

- 功能验收矩阵

## 10. 实现细节要求

## 10.1 状态序列化必须集中管理

不得把仿真前端所需字段散落在：

- `SimulationTab`
- `SimulationViewModel`
- 各子组件
- bridge

之间各写一部分。

必须建立集中序列化层，统一负责：

- 状态字段命名
- 前端 payload 结构
- 结果摘要组装
- 波形视图状态组装
- 详细信息区数据组装

## 10.2 动作桥接只表达用户意图

桥接接口不得直接表达最终 UI 状态。

桥接动作只应该表达：

- 用户要切换什么
- 用户要执行什么
- 用户要请求哪个交互

最终界面状态必须以后端回流的新状态为准。

## 10.3 图形绘制层必须和数据获取层分离

无论最终使用何种前端图形绘制方案，都必须保持：

- 绘制层不负责生成数据
- 绘制层不负责定义波形语义
- 绘制层不负责结果持久化
- 绘制层不负责测量业务计算

## 10.4 添加至对话必须继续走工件路径

前端不得把“添加至对话”退化成把当前页面文本直接塞给对话输入框。

必须继续保留：

- 工件导出
- 文件路径发布
- 对话附件事件流

## 10.5 响应式布局必须是正式设计目标

不能等完成后再用补丁修窄宽度问题。

从一开始就必须明确：

- 顶部概览区如何在窄宽度折叠
- 信号树如何折叠/抽屉化
- 主图和详情区如何重新排布
- 工具栏如何换行
- 表格和日志如何独立滚动

---

## 11. 验收标准

当以下条件同时满足时，视为本轮仿真 Web 化视觉重构完成：

- 仿真显示已经由单一 Web Surface 承载
- 仿真执行、结果存储、撤回机制与显示层边界清晰且未被破坏
- 波形、分析图、分析信息、原始数据、日志、导出、添加至对话能力全部保留
- 框选放大、Fit、光标、测量、信号选择、双轴显示、自适应布局全部正确
- 前端只消费一份统一状态，不存在多套显示真相
- bridge 动作只表达用户意图，不直接操纵 UI DOM 结果
- 旧 Qt 显示壳、旧 tab 结构、旧兼容路径已经被物理删除
- 仓库中不存在“新 Web 前端 + 旧 Qt 页面长期并行”的保留结构

---

## 12. 最终判断

这次重构的正确目标不是“把一个仿真图换成网页图”，而是建立：

- 后端继续保持仿真与结果权威
- 前端用单一 Web 技术栈统一承载显示与交互
- 两者通过单一桥接协议协作
- 旧 Qt 显示体系在 cutover 后被彻底删除

只有这样，最终得到的才会是：

- 视觉统一
- 行为稳定
- 架构清晰
- 可扩展
- 对 AI 和人类开发者都可维护的仿真显示系统
